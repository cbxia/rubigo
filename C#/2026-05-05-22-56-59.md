# Why `Span<T>` Must Live on the Stack

`Span<T>` is declared as a `ref struct`, and that single design decision is the root of everything.

### What is a `ref struct`?

A `ref struct` is a value type with a compiler-enforced constraint: **it can only ever exist on the stack**. The CLR and C# compiler will reject any code that tries to move it to the heap.

---

### The Core Problem It Solves

`Span<T>` is designed to hold a **managed pointer** — a direct reference into memory that could be:

- The interior of a managed heap object (e.g., inside a `string` or `T[]`)
- A stack-allocated buffer (`stackalloc`)
- Native/unmanaged memory

A managed pointer (a `ref T` under the hood) is fundamentally dangerous on the heap because of how the **Garbage Collector (GC)** works.

---

### Why the GC Makes Heap Storage Impossible

The .NET GC is a **compacting** GC. During a collection, it moves live objects to new memory addresses to reduce fragmentation, then updates all references to point to the new locations.

This works fine for normal object references. But `Span<T>` holds an **interior pointer** — a pointer that may point *into the middle of an object*, not to its start. The GC **cannot reliably track and update arbitrary interior pointers** stored inside heap objects. If the GC moved the underlying array but couldn't find and fix the interior pointer inside a heap-allocated `Span<T>`, you'd have a dangling pointer — instant memory corruption.

By restricting `Span<T>` to the stack, the runtime guarantees:
- The GC **never needs to scan or move** a `Span<T>` itself.
- Stack frames have a well-defined, **strictly nested lifetime** — a `Span<T>` cannot outlive the data it points to.

---

### The Specific Restrictions `ref struct` Enforces

Because `Span<T>` is a `ref struct`, the compiler forbids anything that could cause it to escape to the heap:

| Forbidden Operation | Why |
|---|---|
| Boxing (`object o = span`) | Boxing copies to the heap |
| Field in a regular `class` | Class instances live on the heap |
| Field in a non-`ref struct` | That struct could be boxed |
| Use in `async` methods | The state machine is a heap-allocated class |
| Use in lambda/closure captures | Closures are heap-allocated objects |
| Use as a generic type argument | Generics don't guarantee stack allocation |

---

### The Lifetime Safety Guarantee

The stack enforces **strict last-in, first-out (LIFO) lifetimes**. Consider:

```csharp
Span<int> GetSpan()
{
    int[] arr = new int[10]; // heap
    return arr.AsSpan();     // ✅ safe: arr outlives the Span
}

Span<int> GetDangerousSpan()
{
    Span<int> local = stackalloc int[10]; // stack-allocated buffer
    return local; // ❌ compiler error! Would be a dangling pointer
}
```

In the second case, the compiler **catches the escape at compile time** because it knows the `stackalloc` buffer is tied to that stack frame's lifetime. This is enforced through C# 7.2+'s *safe-context* / *ref-safety* rules.

---

### Summary

| Reason | Detail |
|---|---|
| **Compacting GC** | Can't track interior pointers in heap objects |
| **Dangling pointer prevention** | Stack lifetime rules prevent a `Span` outliving its data |
| **`ref struct` constraint** | Compiler statically enforces all escape paths are closed |
| **Zero overhead** | Stack allocation means no GC pressure, no allocation cost |

The stack restriction isn't a limitation — it's the **safety contract** that makes `Span<T>` both fast and correct. It lets you do pointer-level operations with the safety guarantees of managed code.
 
## Further discussions

- Concerning this example of yours:
Span<int> GetSpan()
{
    int[] arr = new int[10]; // heap
    return arr.AsSpan();     // ✅ safe: arr outlives the Span
}

How if the supporting array is GC-compacted/moved before the end of life of the span created on it?

## Excellent Catch — This Exposes a Critical Nuance

The answer is: **the GC updates the `Span<T>`'s internal pointer during compaction, because the `Span<T>` is on the stack** — and the GC knows exactly how to scan the stack.

---

### The GC Can Track Interior Pointers on the Stack

The CLR's JIT compiler emits **GC metadata** for every method — a precise map describing, for each potential GC pause point ("safe point"), exactly which slots on the stack frame contain managed references, including **interior pointers**.

When the GC runs:
1. It suspends threads (or reaches a safe point in concurrent GC)
2. It **walks each thread's stack**, using the JIT's GC info tables
3. It finds every managed reference on every frame — including interior pointers inside `Span<T>` values
4. After moving an object, it **patches all those references** to point to the new location

So if the backing `int[]` gets moved, the `Span<T>` sitting on the stack gets its internal pointer **silently updated** by the GC. From your code's perspective, nothing happened.

---

### So Why Can't `Span<T>` Be on the Heap Then?

This is the precise reason. The GC's ability to track interior pointers **relies on the JIT's stack-frame metadata**. That metadata exists only for stack frames.

For heap objects, the GC tracks references using a completely different mechanism — the **object's type descriptor** (method table), which describes the layout of normal object references within that object. It does **not** have a general mechanism to describe or patch arbitrary interior pointers embedded inside heap objects.

Concretely:

```
Stack frame (JIT metadata exists ✅)
│
├─ Span<int> { ptr → [into heap array], length = 10 }
│         ↑
│         GC knows this is an interior pointer and patches it
│
Heap
├─ int[] at 0x1000  →  moved to 0x5000 by GC
│         ↑
│         Span's ptr is updated: 0x1000 → 0x5000 ✅

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Heap object (no interior-pointer metadata ❌)
│
├─ class Foo { Span<int> span; }   // hypothetical, not allowed
│         ↑
│         GC has NO mechanism to find/patch this interior pointer
│         → dangling pointer after compaction 💥
```

---

### Summary of the Real Rule

| Location of `Span<T>` | GC can patch its interior pointer? | Allowed? |
|---|---|---|
| **Stack** | ✅ Yes — JIT emits precise GC maps per frame | ✅ Yes |
| **Heap** (inside a class) | ❌ No — object layout metadata doesn't cover interior pointers | ❌ Forbidden by `ref struct` |

So your example **is** safe — not *despite* the GC moving the array, but *because* the GC is smart enough to update the `Span`'s pointer when it does. The `ref struct` restriction exists precisely to ensure `Span<T>` **only ever appears in places where that update mechanism works**.

- So how does Memory<T> differs in these aspects discussed on Span<T>? Especially how a Memory<T> typed variable relates to its supporting object differently that allows for more liberty on allocation? The cost?

## `Memory<T>` vs `Span<T>`: The Indirection Trade-off

### The Root Difference: What's Stored Internally

This is the crux of everything. Their internal layouts are fundamentally different:

```
Span<T>  { ref T _reference,  int _length }
            ↑
            Managed interior pointer — directly into the middle of an object

Memory<T> { object _object,  int _index,  int _length }
            ↑
            Normal object reference — points to the START of the object
```

`Memory<T>` **never holds an interior pointer**. It stores a plain object reference (which the GC can track through its normal mechanisms) plus an integer offset. It describes a region *mathematically* rather than pointing into it directly.

---

### How This Enables Heap Allocation

Because `Memory<T>` contains only normal managed references and integers, it's a **regular struct** with no `ref struct` restriction. The GC can handle all its fields through standard object-reference tracking. There are no interior pointers to lose track of.

This means `Memory<T>` can go anywhere:

```csharp
class MyComponent                   // ✅ field in a class (heap)
{
    Memory<byte> _buffer;
}

async Task ProcessAsync()           // ✅ survives across await points
{
    Memory<byte> mem = new byte[1024];
    await SomeIOAsync();            // state machine captures mem on heap
    Use(mem);                       // still valid
}

var list = new List<Memory<char>>;  // ✅ inside a collection
```

None of these are possible with `Span<T>`.

---

### The Cost: Indirection at Access Time

Since `Memory<T>` doesn't hold a direct pointer, **you cannot dereference it directly**. To actually read or write elements, you must call `.Span`, which reconstructs the interior pointer on the stack at that moment:

```csharp
Memory<int> mem = new int[100];

// To do real work, you must materialize a Span:
Span<int> span = mem.Span;   // computes: interior_ptr = &_object[_index]
span[0] = 42;                // direct pointer dereference ✅

// Or via slicing helpers that internally do the same:
mem.Slice(10, 20).Span[0] = 99;
```

The `.Span` property is doing:
1. Unbox/cast `_object` to the concrete array type
2. Compute the interior pointer by adding `_index` as an offset
3. Construct a `Span<T>` on the stack with that pointer and `_length`

This means every "real" access pays:
- A type check / cast on `_object`
- An offset addition
- A `Span<T>` construction

For tight loops processing large buffers this matters, which is why the idiomatic pattern is to **pin** the span once at the boundary of a processing block:

```csharp
void Process(Memory<byte> mem)
{
    Span<byte> span = mem.Span;   // pay the cost once
    for (int i = 0; i < span.Length; i++)
        span[i] ^= 0xFF;          // direct pointer access in the loop
}
```

---

### One More Constraint: No `stackalloc`

Because `Memory<T>` can escape to the heap (e.g., stored in a class field, captured in an async state machine), it **cannot wrap stack-allocated memory**:

```csharp
Span<int>   s = stackalloc int[10];  // ✅ Span is fine — lifetime enforced by stack rules
Memory<int> m = stackalloc int[10];  // ❌ compile error — memory could outlive the stack frame
```

`Memory<T>` is limited to heap-backed sources: arrays, `string`, or a custom `MemoryManager<T>`.

---

### Side-by-Side Summary

| | `Span<T>` | `Memory<T>` |
|---|---|---|
| Internal pointer | Interior pointer (direct) | Object ref + integer offset |
| Struct kind | `ref struct` | Regular `struct` |
| Heap storage | ❌ Forbidden | ✅ Allowed |
| `async` / lambdas | ❌ Forbidden | ✅ Allowed |
| `stackalloc` backing | ✅ Yes | ❌ No |
| Element access cost | Zero — direct dereference | Indirect — must call `.Span` |
| GC interaction | Updated in-place on stack | Normal object-ref tracking |

The design is intentional: `Span<T>` is the **fast, restricted** workhorse for synchronous processing; `Memory<T>` is the **flexible, composable** handle for passing regions of memory across async boundaries, storing them in objects, or putting them in queues — with the understanding that actual work always eventually happens through a `Span<T>`.

- *But anyway fixing span references are more costly than adjusting object references in case memory was  moved*.

## Correct — Interior Pointer Fixup Is More Expensive

You've identified a real asymmetry. The extra cost comes from a fundamental difference in what the GC needs to do when it encounters each type.

---

### Fixing a Normal Object Reference: O(1), Trivial

When an object is moved during compaction, the GC leaves a **forwarding pointer** at the old location. Fixing any normal object reference pointing to it is then:

```
1. Read the reference → old address
2. Read forwarding pointer at old address → new address
3. Write new address back
```

That's it. The reference points to the **start** of the object, and the forwarding pointer is right there at the start. Three memory operations.

---

### Fixing an Interior Pointer: Requires Reverse Lookup

An interior pointer points **somewhere into the middle** of an object. To fix it after the object moves:

```
1. Read the interior pointer → some address in the middle of an object
2. Find which object owns this address  ← the expensive step
3. Compute offset = interior_ptr - object_base
4. Read forwarding pointer at object_base → new_object_base
5. Recompute: new_interior_ptr = new_object_base + offset
6. Write new_interior_ptr back
```

Step 2 is the problem. Given an arbitrary address inside the heap, the GC must **reverse-map it to the owning object's base address**. Objects don't carry a self-referential back-pointer. The GC must use its segment and allocation structures — typically scanning backward through the heap using object size metadata to find the nearest object header before that address. This is fundamentally more work than following a forwarding pointer.

---

### Why .NET Restricts Where Interior Pointers Can Live

This is an additional reason — beyond just the "heap objects have no interior-pointer metadata" argument — why the `ref struct` restriction exists. Each `Span<T>` on the stack is an interior pointer that **must be individually reverse-mapped** during compaction. The cost scales with how many live `Span<T>` values exist at GC time.

Keeping them stack-only contains the problem: stacks are finite in depth, and JIT GC info tables make the scan deterministic and bounded. If `Span<T>` could proliferate across arbitrary heap objects, the number of interior pointers requiring expensive reverse-lookup could grow unboundedly.

---

### This Is Also Why `Memory<T>` Is the Smarter Design for Long-Lived References

Recalling the earlier discussion: `Memory<T>` stores a **normal object reference + integer offset**, never an interior pointer. When the GC moves the backing array, it fixes `Memory<T>._object` via the cheap forwarding-pointer path. The offset stays unchanged. No reverse-lookup needed.

```
Memory<T> fixup:  cheap  — normal object ref → forwarding pointer → done
Span<T>   fixup:  costly — interior ptr → reverse-map to base → offset arithmetic → done
```

This is the deeper reason the `Memory<T>` / `Span<T>` split exists. `Memory<T>` is not just about relaxing lifetime restrictions — it's also the **GC-friendlier representation** for when a region descriptor needs to live beyond a single stack frame.

- *How to represent a collection of spans? For example as the type of return of a function.*

## The Problem: `Span<T>` Cannot Be a Generic Type Argument

Since `Span<T>` is a `ref struct`, the compiler forbids it as a type argument to anything — `List<Span<T>>`, `Span<T>[]`, `IEnumerable<Span<T>>` are all illegal. You need a different strategy depending on the lifetime requirements.

---

### Option 1: Return `Memory<T>` Instead

The most straightforward solution. Since `Memory<T>` is a regular struct, it composes freely:

```csharp
Memory<byte>[]              GetSegments();
IReadOnlyList<Memory<byte>> GetSegments();
List<Memory<byte>>          GetSegments();
```

The caller materializes a `Span<T>` only at the point of actual processing:

```csharp
foreach (Memory<byte> segment in GetSegments())
{
    Span<byte> span = segment.Span;   // pay indirection cost once per segment
    Process(span);
}
```

**Cost:** the `Memory<T>` indirection on every access, plus the array/list allocation.

---

### Option 2: Single Buffer + Ranges (Most Efficient)

Instead of a collection of spans, return one contiguous block of data and an array of `Range` (or index pairs) describing where each segment is. The caller slices on demand:

```csharp
(Memory<byte> buffer, Range[] segments) Tokenize(ReadOnlySpan<byte> input)
{
    var buffer   = new byte[input.Length].AsMemory();
    var segments = new List<Range>();

    // fill buffer, record segment ranges...

    return (buffer, segments.ToArray());
}

// Caller:
var (buf, segs) = Tokenize(data);
foreach (Range r in segs)
{
    Span<byte> span = buf.Span[r];   // zero-copy slice
    Process(span);
}
```

**Cost:** two allocations (buffer + ranges array), but no per-segment object overhead. Slicing is free.

---

### Option 3: Custom `ref struct` with a `Span<Range>` (Stack-Only, Zero Allocation)

When everything stays on the stack — synchronous, no async, no escaping — you can represent the collection entirely without heap allocation:

```csharp
ref struct SegmentCollection
{
    private readonly Span<byte>  _buffer;
    private readonly Span<Range> _segments;
    private int _count;

    public SegmentCollection(Span<byte> buffer, Span<Range> segmentStorage)
    {
        _buffer   = buffer;
        _segments = segmentStorage;
        _count    = 0;
    }

    public void Add(Range r) => _segments[_count++] = r;

    public Span<byte> this[int i] => _buffer[_segments[i]];
    public int Count => _count;
}

SegmentCollection Tokenize(Span<byte> input, Span<Range> rangeStorage)
{
    var result = new SegmentCollection(input, rangeStorage);
    // ... fill result ...
    return result;
}

// Caller — entirely stack allocated:
Span<byte>  buf     = stackalloc byte[4096];
Span<Range> storage = stackalloc Range[64];

SegmentCollection segments = Tokenize(buf, storage);
for (int i = 0; i < segments.Count; i++)
    Process(segments[i]);
```

**Cost:** zero heap allocations. The caller provides the storage buffers, which is the idiomatic pattern in high-performance .NET (the "destination buffer" pattern used throughout `System.IO.Pipelines`).

---

### Option 4: `System.Buffers.ReadOnlySequence<T>` (For Discontiguous Data)

For truly discontiguous data (e.g., data arriving in network chunks), the BCL provides `ReadOnlySequence<T>` — a linked-list of `ReadOnlyMemory<T>` segments designed exactly for this problem:

```csharp
ReadOnlySequence<byte> GetData();

// Caller:
foreach (ReadOnlyMemory<byte> segment in GetData())
{
    ReadOnlySpan<byte> span = segment.Span;
    Process(span);
}
```

This is what `System.IO.Pipelines.PipeReader` returns — the standard abstraction for streaming discontiguous byte data in .NET.

---

### Decision Guide

| Scenario | Approach |
|---|---|
| General purpose, may cross async | `Memory<T>[]` or `IReadOnlyList<Memory<T>>` |
| Contiguous data, performance matters | Single buffer + `Range[]` |
| Hot path, synchronous, zero-alloc | Custom `ref struct` + caller-provided `Span<Range>` |
| Discontiguous / streaming data | `ReadOnlySequence<T>` |

The recurring theme is: **`Span<T>` is always the leaf-level access type** — the thing you process with. The collection, however it's represented, is always built from `Memory<T>`, ranges, or higher-level abstractions.
