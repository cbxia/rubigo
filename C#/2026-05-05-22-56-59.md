## Why `Span<T>` Must Live on the Stack

`Span<T>` is declared as a `ref struct`, and that single design decision is the root of everything.

### What is a `ref struct`?

A `ref struct` is a value type with a compiler-enforced constraint: **it can only ever exist on the stack**. The CLR and C# compiler will reject any code that tries to move it to the heap.

---

### The Core Problem It Solves

`Span<T>` is designed to hold a **managed pointer** — a direct reference into memory that could be:

- The interior of a managed heap object (e.g., inside a `string` or `T[]`)
- A stack-allocated buffer (`stackalloc`)
- Native/unmanaged memory

A managed pointer (a `ref T` under the hood) is fundamentally dangerous on the heap because of how the **Garbage Collector (GC)** works.

---

### Why the GC Makes Heap Storage Impossible

The .NET GC is a **compacting** GC. During a collection, it moves live objects to new memory addresses to reduce fragmentation, then updates all references to point to the new locations.

This works fine for normal object references. But `Span<T>` holds an **interior pointer** — a pointer that may point *into the middle of an object*, not to its start. The GC **cannot reliably track and update arbitrary interior pointers** stored inside heap objects. If the GC moved the underlying array but couldn't find and fix the interior pointer inside a heap-allocated `Span<T>`, you'd have a dangling pointer — instant memory corruption.

By restricting `Span<T>` to the stack, the runtime guarantees:
- The GC **never needs to scan or move** a `Span<T>` itself.
- Stack frames have a well-defined, **strictly nested lifetime** — a `Span<T>` cannot outlive the data it points to.

---

### The Specific Restrictions `ref struct` Enforces

Because `Span<T>` is a `ref struct`, the compiler forbids anything that could cause it to escape to the heap:

| Forbidden Operation | Why |
|---|---|
| Boxing (`object o = span`) | Boxing copies to the heap |
| Field in a regular `class` | Class instances live on the heap |
| Field in a non-`ref struct` | That struct could be boxed |
| Use in `async` methods | The state machine is a heap-allocated class |
| Use in lambda/closure captures | Closures are heap-allocated objects |
| Use as a generic type argument | Generics don't guarantee stack allocation |

---

### The Lifetime Safety Guarantee

The stack enforces **strict last-in, first-out (LIFO) lifetimes**. Consider:

```csharp
Span<int> GetSpan()
{
    int[] arr = new int[10]; // heap
    return arr.AsSpan();     // ✅ safe: arr outlives the Span
}

Span<int> GetDangerousSpan()
{
    Span<int> local = stackalloc int[10]; // stack-allocated buffer
    return local; // ❌ compiler error! Would be a dangling pointer
}
```

In the second case, the compiler **catches the escape at compile time** because it knows the `stackalloc` buffer is tied to that stack frame's lifetime. This is enforced through C# 7.2+'s *safe-context* / *ref-safety* rules.

---

### Summary

| Reason | Detail |
|---|---|
| **Compacting GC** | Can't track interior pointers in heap objects |
| **Dangling pointer prevention** | Stack lifetime rules prevent a `Span` outliving its data |
| **`ref struct` constraint** | Compiler statically enforces all escape paths are closed |
| **Zero overhead** | Stack allocation means no GC pressure, no allocation cost |

The stack restriction isn't a limitation — it's the **safety contract** that makes `Span<T>` both fast and correct. It lets you do pointer-level operations with the safety guarantees of managed code.
