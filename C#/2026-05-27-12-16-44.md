## 1. The Allocation Problem in High-Throughput Telemetry

Network telemetry systems ingest continuous streams of measurement data — metrics, spans, events, and counters — often at rates of hundreds of thousands of frames per second per connection. A single monitoring agent aggregating data from dozens of microservices can easily process millions of small messages per minute.

The naive approach — `StreamReader.ReadLineAsync()` followed by `string.Split()` — creates an avalanche of heap allocations for every frame:

```csharp
// ❌ The naive approach: allocates on every single frame
while ((string? line = await reader.ReadLineAsync()) != null)
{
    // Each ReadLineAsync: allocates a new string
    // Each Split: allocates a string[] + N substrings
    string[] parts = line.Split('|');
    
    var metric = new TelemetryMetric
    {
        Name      = parts[0],           // substring allocation
        Timestamp = long.Parse(parts[1]), // string → long: temp string consumed
        Value     = double.Parse(parts[2])
    };
    
    await ProcessMetricAsync(metric);
}
```

For a system receiving 500,000 frames/sec, even a modest 200 bytes of garbage per frame generates **100 MB/sec of GC pressure**. Gen 0 collections begin to spike. Latency becomes non-deterministic. The GC pauses arrive precisely when your monitoring system needs to be most responsive to production incidents — a catastrophic irony.

### What "Zero-Allocation" Actually Means

Zero-allocation parsing doesn't mean zero memory use. It means:

- **No heap allocations per-frame** for parsing machinery — only for the final parsed object if it must escape the stack
- **No string intermediaries** for numeric conversion
- **No array copying** when slicing protocol fields
- **Rented buffers** returned predictably, not abandoned to the GC

The .NET runtime provides three layered APIs to achieve this:

| API | Purpose |
|-----|---------|
| `System.Memory<T>` / `ReadOnlyMemory<T>` | Managed heap, stack, or native memory as a uniform abstraction |
| `System.Span<T>` / `ReadOnlySpan<T>` | Stack-only, fast, slice-and-parse without copying |
| `System.IO.Pipelines` | Backpressure-aware, zero-copy I/O over any transport |

## 2. The Pipeline Model: Rethinking I/O

`System.IO.Pipelines`, introduced in .NET Core 2.1, fundamentally changes how you consume streaming data. Instead of reading into a caller-managed buffer, the pipeline owns buffer management internally, exposing data through a `PipeReader` interface.

### The Classic Stream Problem

The fundamental issue with `Stream.ReadAsync()` is that callers must pre-allocate a buffer and then deal with *partial reads*:

```csharp
// ❌ Classic stream: partial reads require complex state machines
byte[] buffer = new byte[4096];
int consumed = 0;

while (true)
{
    int bytesRead = await stream.ReadAsync(buffer, consumed, buffer.Length - consumed);
    if (bytesRead == 0) break;
    consumed += bytesRead;
    
    // Now: where does one frame end and the next begin?
    // What if the buffer is too small for a frame?
    // What if we read half a frame? We must track the leftover manually.
    int frameEnd = FindFrameEnd(buffer, 0, consumed);
    if (frameEnd < 0)
    {
        // Incomplete frame — wait for more data, possibly resize buffer
        // ...manual bookkeeping nightmare begins
    }
}
```

### The Pipeline Solution

`PipeReader` separates *reading* from *consuming*. The reader calls `TryRead` or `ReadAsync` to get a `ReadResult` containing a `ReadOnlySequence<byte>` — all the buffered data available. After examining it, the reader tells the pipe exactly how much data was *examined* (advanced past in the search) and how much was *consumed* (can be discarded from the buffer):

```
   PipeWriter ──[produces]──► Pipe Buffer ──[exposes]──► PipeReader
                                                              │
                                            ┌─────────────────┤
                                            │                 │
                                       AdvanceTo(            │
                                         consumed: ◄─── what we parsed
                                         examined: ◄─── how far we looked
                                       )
```

This means:
- **No buffer resizing**: the pipe grows its internal linked-list of buffer segments automatically
- **No leftover tracking**: unconsumed data stays in the pipe; the reader sees it on the next iteration
- **Backpressure is built-in**: the pipe pauses the writer when buffered data exceeds `PauseWriterThreshold`

## 3. Protocol Design: A Binary Telemetry Frame

For this article, we implement a parser for a compact binary telemetry protocol with both fixed-width binary frames and newline-delimited text frames (common in real-world systems that mix human-readable and machine-optimized formats).

### Binary Frame Layout

```
┌─────────────────────────────────────────────────────────────────────┐
│  Magic  │  Ver  │FrameType│ MetricID │   Timestamp   │    Value    │
│ 2 bytes │ 1 byte│ 1 byte  │ 4 bytes  │   8 bytes     │   8 bytes   │
│  0xA1B2 │  0x01 │ 0x01=   │ uint32   │  int64 (Unix  │  double     │
│         │       │ Gauge,  │          │  nanoseconds) │  IEEE 754   │
│         │       │ 0x02=   │          │               │             │
│         │       │ Counter │          │               │             │
└─────────────────────────────────────────────────────────────────────┘
Total: 24 bytes per binary frame
```

### Text Frame Layout (line-delimited fallback)

```
METRIC|<name>|<timestamp_ns>|<value>\n

Example:
METRIC|cpu.usage.pct|1716729600000000000|72.45\n
```

### Protocol Constants

```csharp
internal static class TelemetryProtocol
{
    // Binary frame
    public const ushort MagicNumber      = 0xA1B2;
    public const byte   ProtocolVersion  = 0x01;
    public const int    BinaryFrameSize  = 24;
    
    public const byte   FrameTypeGauge   = 0x01;
    public const byte   FrameTypeCounter = 0x02;
    public const byte   FrameTypeText    = 0x00; // signals text fallback
    
    // Text frame
    public static readonly byte[] TextPrefix   = "METRIC|"u8.ToArray();
    public const byte              FieldDelimiter = (byte)'|';
    public const byte              LineDelimiter  = (byte)'\n';
    
    // Offsets within binary frame
    public const int OffsetMagic     = 0;
    public const int OffsetVersion   = 2;
    public const int OffsetFrameType = 3;
    public const int OffsetMetricId  = 4;
    public const int OffsetTimestamp = 8;
    public const int OffsetValue     = 16;
}
```

## 5. Core Primitives: Spans, Memory, and Sequences

Before the parser, you must deeply understand the three memory abstractions in play.

### `Span<T>` and `ReadOnlySpan<T>`

`Span<T>` is a stack-only ref struct that represents a contiguous region of typed memory — an array segment, a stack allocation, or a native buffer — without copying. Because it is stack-bound, it cannot be stored on the heap or used in async methods directly.

```csharp
// All three point to memory without any heap allocation:
Span<byte> fromArray  = new byte[64];
Span<byte> fromStack  = stackalloc byte[64];
Span<byte> fromSlice  = someArray.AsSpan(offset: 10, length: 24);

// Slicing is O(1) — just adjusting a pointer + length:
ReadOnlySpan<byte> header = fromSlice[..4];  // first 4 bytes, zero-copy
ReadOnlySpan<byte> body   = fromSlice[4..];  // rest, zero-copy
```

### `Memory<T>` and `ReadOnlyMemory<T>`

`Memory<T>` is the heap-safe counterpart. It can be stored in fields, passed across `await` boundaries, and stored in collections. Under the hood it carries an object reference + offset + length:

```csharp
// Memory<T> can cross async boundaries:
Memory<byte>         mem       = new byte[1024];
ReadOnlyMemory<byte> readOnlyMem = mem;

// Convert to Span only within a synchronous scope:
ReadOnlySpan<byte> span = readOnlyMem.Span; // valid here
await Task.Delay(1);
// span is invalid after any await — use readOnlyMem instead
```

### `ReadOnlySequence<byte>`: The Pipeline's Data Model

`ReadOnlySequence<byte>` is the most important type for pipeline parsing. The `PipeReader` exposes all buffered data as a single `ReadOnlySequence<byte>`, which may internally span *multiple non-contiguous memory segments* — a linked list of `ReadOnlyMemory<byte>` chunks.

```
ReadOnlySequence<byte> (logical view: one contiguous byte stream)
│
├─► Segment 0:  [A A A A A A A A]   (8 bytes in ArrayPool chunk #1)
├─► Segment 1:  [B B B B B B]       (6 bytes in ArrayPool chunk #2)
└─► Segment 2:  [C C C C]           (4 bytes in ArrayPool chunk #3)

Total length: 18 bytes — but NOT contiguous in memory
```

This non-contiguity is the central challenge of pipeline parsing. A frame may straddle a segment boundary. Naively calling `.ToArray()` on a sequence collapses it into a new contiguous array — defeating zero-allocation goals. The correct approach is to either:

1. **Work with individual segments** using `foreach` over the sequence's enumerator
2. **Check `IsSingleSegment`** and fast-path to `.FirstSpan` when possible
3. **Use `SequenceReader<byte>`** — a cursor-based reader that handles multi-segment sequences transparently

```csharp
// Checking for the fast path:
if (buffer.IsSingleSegment)
{
    // ✅ No copy needed — direct span access
    ReadOnlySpan<byte> span = buffer.FirstSpan;
    ParseFromSpan(span);
}
else
{
    // Multi-segment: use SequenceReader<byte> or copy a bounded amount
    var reader = new SequenceReader<byte>(buffer);
    ParseFromReader(ref reader);
}
```

## 6. Building the PipeReader Parser

The top-level parser hosts the `PipeReader` and orchestrates the parsing loop. The key design principles:

- **Never `await` while holding a `Span`** — spans cannot cross async boundaries
- **Always call `AdvanceTo` exactly once per `ReadAsync` call**, even on error
- **Distinguish `examined` from `consumed`** — they are not the same
- **Respect `IsCompleted`** — signal from the writer that no more data is coming

```csharp
using System.Buffers;
using System.IO.Pipelines;
using Microsoft.Extensions.Logging;

namespace TelemetryParser.Parsing;

/// <summary>
/// Processes a continuous stream of binary/text telemetry frames from a PipeReader.
/// Designed for zero heap allocation per frame in the hot path.
/// </summary>
public sealed class TelemetryPipeParser
{
    private readonly ILogger<TelemetryPipeParser> _logger;
    private readonly ITelemetryFrameHandler       _handler;
    private long _framesProcessed;
    private long _parseErrors;

    public TelemetryPipeParser(
        ILogger<TelemetryPipeParser> logger,
        ITelemetryFrameHandler handler)
    {
        _logger  = logger;
        _handler = handler;
    }

    public long FramesProcessed => _framesProcessed;
    public long ParseErrors     => _parseErrors;

    /// <summary>
    /// Reads frames from the pipe until the writer completes or cancellation is requested.
    /// </summary>
    public async ValueTask ParseAsync(PipeReader reader, CancellationToken ct = default)
    {
        try
        {
            while (!ct.IsCancellationRequested)
            {
                // ── Step 1: Await data from the pipe ──────────────────────────────
                ReadResult result = await reader.ReadAsync(ct);
                ReadOnlySequence<byte> buffer = result.Buffer;

                // ── Step 2: Process all complete frames in the buffer (sync) ──────
                // This loop is synchronous — no allocations, no async overhead.
                SequencePosition consumed = buffer.Start;
                SequencePosition examined = buffer.Start;

                try
                {
                    while (TryParseNextFrame(ref buffer, out TelemetryFrame frame,
                                             out consumed, out examined))
                    {
                        // Hand off to handler — may be async, but we've already
                        // parsed the frame into a value type (stack-resident).
                        await _handler.HandleAsync(frame, ct);
                        Interlocked.Increment(ref _framesProcessed);
                    }
                }
                catch (ProtocolException ex)
                {
                    // Log and skip the offending byte to recover sync
                    _logger.LogWarning(ex, "Protocol error after {Frames} frames", _framesProcessed);
                    Interlocked.Increment(ref _parseErrors);
                    
                    // Advance past one byte to attempt resynchronization
                    consumed = buffer.GetPosition(1, buffer.Start);
                    examined = consumed;
                }
                finally
                {
                    // ── Step 3: ALWAYS advance the reader ─────────────────────────
                    // consumed: data the pipe may discard
                    // examined: how far we looked (pipe won't re-signal until new data arrives past here)
                    reader.AdvanceTo(consumed, examined);
                }

                // ── Step 4: Check for completion ──────────────────────────────────
                if (result.IsCompleted)
                {
                    if (buffer.Length > 0)
                    {
                        _logger.LogWarning(
                            "Pipe completed with {Bytes} bytes of incomplete frame data", 
                            buffer.Length);
                    }
                    break;
                }
            }
        }
        finally
        {
            await reader.CompleteAsync();
        }
    }

    /// <summary>
    /// Attempts to parse exactly one frame from the buffer.
    /// Returns false if insufficient data is present (caller should wait for more).
    /// </summary>
    private static bool TryParseNextFrame(
        ref ReadOnlySequence<byte> buffer,
        out TelemetryFrame frame,
        out SequencePosition consumed,
        out SequencePosition examined)
    {
        frame    = default;
        consumed = buffer.Start;
        examined = buffer.End; // Optimistic: we examined everything available

        if (buffer.IsEmpty)
            return false;

        // Peek at the first two bytes to determine frame type
        // (magic number for binary, 'M' for text "METRIC|...")
        Span<byte> header = stackalloc byte[2];
        if (!TryCopyExact(buffer, header))
        {
            // Not even 2 bytes available — wait for more data.
            // examined stays at buffer.End so the pipe knows we need more.
            return false;
        }

        bool isBinary = BinaryPrimitives_ReadUInt16BigEndian(header) == TelemetryProtocol.MagicNumber;

        if (isBinary)
        {
            return BinaryFrameParser.TryParse(ref buffer, out frame, out consumed, out examined);
        }
        else
        {
            return TextFrameParser.TryParse(ref buffer, out frame, out consumed, out examined);
        }
    }

    // Helper: avoids System.Buffers.Binary reference at call site
    private static ushort BinaryPrimitives_ReadUInt16BigEndian(ReadOnlySpan<byte> source)
        => (ushort)((source[0] << 8) | source[1]);

    /// <summary>
    /// Copies exactly <paramref name="destination"/>.Length bytes from the sequence start.
    /// Returns false if insufficient bytes are available (does not throw).
    /// Does NOT advance the sequence.
    /// </summary>
    private static bool TryCopyExact(in ReadOnlySequence<byte> buffer, Span<byte> destination)
    {
        if (buffer.Length < destination.Length)
            return false;
        
        buffer.Slice(0, destination.Length).CopyTo(destination);
        return true;
    }
}
```

## 7. Slicing ReadOnlySequence\<byte\>

Slicing is where many implementations go wrong, either by calling `.ToArray()` unnecessarily or by failing to handle multi-segment sequences.

        // Fast path: the entire frame is in one co
