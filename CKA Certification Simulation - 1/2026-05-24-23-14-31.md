**Question 1**
> **Weight: 8% | Estimated time: 15 min**

You are required to initialize a new Kubernetes control plane node using `kubeadm`. The cluster must meet the following requirements:

- API server advertise address: `192.168.1.10`
- Pod network CIDR: `10.244.0.0/16`
- Kubernetes version: `1.30.0`
- The control plane endpoint DNS name: `k8s-cp.lab.internal`

After initialization, configure `kubectl` for the `root` user and install the **Flannel** CNI plugin.

**Tasks:**
1. Run the appropriate `kubeadm init` command.
2. Set up `kubeconfig` for root.
3. Apply the Flannel CNI manifest.

<details><summary>💡 Answer</summary>

```bash
# 1. Initialize the control plane
kubeadm init \
  --apiserver-advertise-address=192.168.1.10 \
  --pod-network-cidr=10.244.0.0/16 \
  --control-plane-endpoint=k8s-cp.lab.internal \
  --kubernetes-version=1.30.0

# 2. Configure kubectl for root
mkdir -p $HOME/.kube
cp /etc/kubernetes/admin.conf $HOME/.kube/config
chown $(id -u):$(id -g) $HOME/.kube/config

# 3. Apply Flannel CNI
kubectl apply -f https://raw.githubusercontent.com/flannel-io/flannel/master/Documentation/kube-flannel.yml
```

</details>

**Question 2**
> **Weight: 6% | Estimated time: 10 min**

A worker node `worker-node-02` has been provisioned. Join it to an existing cluster. The join token has expired.

**Tasks:**
1. On the **control plane node**, generate a new bootstrap token valid for 2 hours.
2. Print the full `kubeadm join` command including the CA cert hash.
3. Execute the join on `worker-node-02`.

<details><summary>💡 Answer</summary>

```bash
# 1 & 2. On the control plane — generate token and print join command
kubeadm token create --ttl 2h --print-join-command

# Output will look like:
# kubeadm join 192.168.1.10:6443 \
#   --token <token> \
#   --discovery-token-ca-cert-hash sha256:<hash>

# Alternative: manually compute the CA hash
openssl x509 -pubkey -in /etc/kubernetes/pki/ca.crt | \
  openssl rsa -pubin -outform der 2>/dev/null | \
  openssl dgst -sha256 -hex | sed 's/^.* //'

# 3. On worker-node-02, run the printed join command
kubeadm join 192.168.1.10:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash>
```

</details>

**Question 3**
> **Weight: 7% | Estimated time: 12 min**

The `kubelet` on node `worker-node-01` is not starting. The node shows `NotReady`.

**Tasks:**
1. Identify the root cause using `systemd` and `journalctl`.
2. The kubelet config file must explicitly set the `cgroupDriver` to `systemd`.
3. Restart and verify the kubelet is active.

<details><summary>💡 Answer</summary>

```bash
# 1. Check kubelet status and logs
systemctl status kubelet
journalctl -xeu kubelet --no-pager | tail -50

# 2. Edit kubelet config (adjust path if needed)
vi /var/lib/kubelet/config.yaml
# Ensure this line is present:
# cgroupDriver: systemd

# Or patch via the kubeadm drop-in:
vi /etc/systemd/system/kubelet.service.d/10-kubeadm.conf
# Add: --cgroup-driver=systemd to KUBELET_EXTRA_ARGS if needed

# 3. Reload and restart
systemctl daemon-reload
systemctl restart kubelet
systemctl status kubelet

# Verify node status from control plane
kubectl get nodes
```

</details>

**Question 4**
> **Weight: 5% | Estimated time: 8 min**

Configure `kubeadm` to use an **external etcd** cluster instead of the default stacked etcd. The etcd endpoint is `https://etcd-01.lab.internal:2379`.

Relevant certificates are located at:
- CA: `/etc/etcd/pki/ca.crt`
- Client cert: `/etc/etcd/pki/apiserver-etcd-client.crt`
- Client key: `/etc/etcd/pki/apiserver-etcd-client.key`

**Task:** Write the `kubeadm` configuration YAML that uses this external etcd.

<details><summary>💡 Answer</summary>

```yaml
# kubeadm-config.yaml
apiVersion: kubeadm.k8s.io/v1beta3
kind: ClusterConfiguration
kubernetesVersion: "1.30.0"
controlPlaneEndpoint: "k8s-cp.lab.internal:6443"
etcd:
  external:
    endpoints:
      - "https://etcd-01.lab.internal:2379"
    caFile: "/etc/etcd/pki/ca.crt"
    certFile: "/etc/etcd/pki/apiserver-etcd-client.crt"
    keyFile: "/etc/etcd/pki/apiserver-etcd-client.key"
```

```bash
# Then initialize with:
kubeadm init --config=kubeadm-config.yaml
```
</details>

**Question 5**
> **Weight: 6% | Estimated time: 10 min**

Upgrade the control plane node from Kubernetes `1.29.0` to `1.30.0` using `kubeadm`.

**Tasks:**
1. Drain the control plane node safely.
2. Upgrade `kubeadm`, then apply the upgrade plan.
3. Upgrade `kubelet` and `kubectl`.
4. Uncordon the node.

<details><summary>💡 Answer</summary>

```bash
# 1. Drain control plane (allow daemonsets, ignore standalone pods)
kubectl drain <control-plane-node> \
  --ignore-daemonsets \
  --delete-emptydir-data

# 2. Upgrade kubeadm
apt-mark unhold kubeadm
apt-get install -y kubeadm=1.30.0-00
apt-mark hold kubeadm

kubeadm upgrade plan
kubeadm upgrade apply v1.30.0

# 3. Upgrade kubelet and kubectl
apt-mark unhold kubelet kubectl
apt-get install -y kubelet=1.30.0-00 kubectl=1.30.0-00
apt-mark hold kubelet kubectl

systemctl daemon-reload
systemctl restart kubelet

# 4. Uncordon
kubectl uncordon <control-plane-node>
kubectl get nodes
```

</details>

**Question 6**
> **Weight: 9% | Estimated time: 15 min**

Take a **snapshot backup** of the etcd cluster. The etcd is running as a static pod on the control plane.

Connection details:
- Endpoint: `https://127.0.0.1:2379`
- CA cert: `/etc/kubernetes/pki/etcd/ca.crt`
- Server cert: `/etc/kubernetes/pki/etcd/server.crt`
- Server key: `/etc/kubernetes/pki/etcd/server.key`

**Task:** Save the snapshot to `/opt/backup/etcd-snapshot-$(date +%Y%m%d).db` and verify its integrity.

<details><summary>💡 Answer</summary>

```bash
# Create backup directory
mkdir -p /opt/backup

# Take snapshot
ETCDCTL_API=3 etcdctl snapshot save \
  /opt/backup/etcd-snapshot-$(date +%Y%m%d).db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Verify snapshot integrity
ETCDCTL_API=3 etcdctl snapshot status \
  /opt/backup/etcd-snapshot-$(date +%Y%m%d).db \
  --write-out=table
```

Expected output includes: `hash`, `revision`, `total keys`, `total size`.
</details>

**Question 7**
> **Weight: 10% | Estimated time: 18 min**

The cluster has suffered data corruption. **Restore etcd** from a previously taken snapshot located at `/opt/backup/etcd-snapshot-20240501.db`.

The etcd data directory is `/var/lib/etcd`.

**Tasks:**
1. Stop the API server.
2. Restore the snapshot to a new data directory.
3. Update the etcd static pod manifest to point to the new directory.
4. Verify the cluster recovers.

<details><summary>💡 Answer</summary>

```bash
# 1. Move the API server manifest to stop it (static pod)
mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/
mv /etc/kubernetes/manifests/etcd.yaml /tmp/

# 2. Restore snapshot to a new directory
ETCDCTL_API=3 etcdctl snapshot restore \
  /opt/backup/etcd-snapshot-20240501.db \
  --data-dir=/var/lib/etcd-restore \
  --name=master \
  --initial-cluster=master=https://127.0.0.1:2380 \
  --initial-advertise-peer-urls=https://127.0.0.1:2380

# 3. Update the etcd manifest to use new data directory
# Edit the hostPath in etcd.yaml:
sed -i 's|/var/lib/etcd|/var/lib/etcd-restore|g' /tmp/etcd.yaml

# Fix ownership
chown -R etcd:etcd /var/lib/etcd-restore

# 4. Restore manifests
mv /tmp/etcd.yaml /etc/kubernetes/manifests/
mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

# Wait and verify
sleep 30
kubectl get nodes
kubectl get pods -A
```

</details>

**Question 8**
> **Weight: 7% | Estimated time: 12 min**

You need to inspect the current state of the etcd cluster and perform health checks.

**Tasks:**
1. Check the health of all etcd members.
2. List all members and identify the current **leader**.
3. Display the total number of keys stored in etcd.

<details><summary>💡 Answer</summary>

```bash
# Set common flags as env vars for convenience
export ETCDCTL_API=3
export ETCDCTL_ENDPOINTS=https://127.0.0.1:2379
export ETCDCTL_CACERT=/etc/kubernetes/pki/etcd/ca.crt
export ETCDCTL_CERT=/etc/kubernetes/pki/etcd/server.crt
export ETCDCTL_KEY=/etc/kubernetes/pki/etcd/server.key

# 1. Health check
etcdctl endpoint health

# 2. List members and identify leader
etcdctl member list --write-out=table
etcdctl endpoint status --write-out=table
# The "IS LEADER" column identifies the leader

# 3. Count total keys
etcdctl get / --prefix --keys-only | grep -v '^$' | wc -l
```

</details>

**Question 9**
> **Weight: 8% | Estimated time: 14 min**

Add a **new etcd member** to an existing etcd cluster to expand it from 1 to 3 members (for HA). The new member will run on `etcd-02.lab.internal` with peer URL `https://192.168.1.21:2380`.

**Tasks:**
1. Register the new member from the existing etcd node.
2. Provide the correct `--initial-cluster` and `--initial-cluster-state` flags the new member must use when starting.

<details><summary>💡 Answer</summary>

```bash
# 1. On the EXISTING etcd node — add the new member
ETCDCTL_API=3 etcdctl member add etcd-02 \
  --peer-urls=https://192.168.1.21:2380 \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# 2. The command output will include the flags needed.
# Start the new etcd member on etcd-02 with:
etcd \
  --name=etcd-02 \
  --initial-advertise-peer-urls=https://192.168.1.21:2380 \
  --listen-peer-urls=https://192.168.1.21:2380 \
  --listen-client-urls=https://192.168.1.21:2379,https://127.0.0.1:2379 \
  --advertise-client-urls=https://192.168.1.21:2379 \
  --initial-cluster=etcd-01=https://192.168.1.20:2380,etcd-02=https://192.168.1.21:2380 \
  --initial-cluster-state=existing \   # ← critical: NOT "new"
  --data-dir=/var/lib/etcd \
  --cert-file=/etc/etcd/pki/server.crt \
  --key-file=/etc/etcd/pki/server.key \
  --trusted-ca-file=/etc/etcd/pki/ca.crt \
  --peer-cert-file=/etc/etcd/pki/peer.crt \
  --peer-key-file=/etc/etcd/pki/peer.key \
  --peer-trusted-ca-file=/etc/etcd/pki/ca.crt

# Verify
etcdctl member list --write-out=table
```

</details>

**Question 10**
> **Weight: 9% | Estimated time: 15 min**

Your etcd cluster is running but you suspect it has **performance issues** due to excessive historical revision data.

**Tasks:**
1. Check the current etcd DB size and compaction revision.
2. Compact etcd up to the **latest revision** to reclaim space.
3. Run **defragmentation** on all etcd endpoints.
4. Verify the DB size has decreased.

<details><summary>💡 Answer</summary>

```bash
export ETCDCTL_API=3
export ETCDCTL_ENDPOINTS=https://127.0.0.1:2379
export ETCDCTL_CACERT=/etc/kubernetes/pki/etcd/ca.crt
export ETCDCTL_CERT=/etc/kubernetes/pki/etcd/server.crt
export ETCDCTL_KEY=/etc/kubernetes/pki/etcd/server.key

# 1. Check DB size and current revision
etcdctl endpoint status --write-out=table
# Note the "DB SIZE" and "RAFT TERM" columns

# Get the current revision number
REV=$(etcdctl endpoint status --write-out=json | \
  python3 -c "import sys,json; \
  print(json.load(sys.stdin)[0]['Status']['header']['revision'])")

echo "Current revision: $REV"

# 2. Compact up to the latest revision
etcdctl compact $REV

# 3. Defragment all endpoints
etcdctl defrag --cluster

# 4. Verify size decreased
etcdctl endpoint status --write-out=table
```

> ⚠️ **Key exam tip:** Always compact **before** defragmenting. Defrag alone does not remove historical data — compaction must happen first.
</details>

## 📋 Quick Reference Cheat Sheet

| Topic | Key Command |
|---|---|
| Init cluster | `kubeadm init --pod-network-cidr --apiserver-advertise-address` |
| Generate join cmd | `kubeadm token create --print-join-command` |
| etcd backup | `etcdctl snapshot save <path> --endpoints --cacert --cert --key` |
| etcd restore | `etcdctl snapshot restore <path> --data-dir` |
| etcd health | `etcdctl endpoint health` |
| etcd leader | `etcdctl endpoint status --write-out=table` |
| Compact | `etcdctl compact <rev>` |
| Defrag | `etcdctl defrag --cluster` |
| Upgrade | `kubeadm upgrade apply v1.x.x` |

> 💡 **Exam Tips:** Always set `ETCDCTL_API=3`. Use `--write-out=table` for readable output. Check static pod manifests in `/etc/kubernetes/manifests/` for etcd configuration.
